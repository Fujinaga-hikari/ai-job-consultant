# Vercel 前提リリース ToDo

採用担当者向け LP（AI 求人ドラフト生成 → 掲載サービス誘導）を **Vercel 上で運用する**ためのタスク一覧。  
DB は **Vercel Postgres**（方針確定）。メールは **Resend**（契約済み想定）。AI は **Gemini**（サーバー側のみ）。

---

## 1. アプリ／リポジトリ

- [ ] Next.js（App Router）＋ API Routes で再構築（既存 `app.py` Streamlit は置き換え前提）
- [ ] 本番ビルドが通る状態（`package.json` / `pnpm` or `npm`）
- [ ] `README.md`：ローカル起動、環境変数、`vercel` デプロイ、DB マイグレーション手順
- [ ] `.env.example`：`GEMINI_API_KEY`、`POSTGRES_URL`（または Vercel が注入する変数名）、`RESEND_API_KEY`（任意）、Slack Webhook 等（任意）

---

## 2. Vercel プロジェクト

- [ ] GitHub リポジトリと Vercel を連携
- [ ] **本番・プレビュー**の環境変数を設定（シークレットはダッシュボード / CLI）
- [ ] Node バージョンがプロジェクトと一致していることの確認

---

## 3. データベース（Vercel Postgres）

- [ ] Vercel 上で Postgres を作成し、プロジェクトに接続
- [ ] ORM（Drizzle / Prisma 等）とマイグレーションをリポジトリに含める
- [ ] スキーマ案（例）
  - リード：メール、同意フラグ、任意の氏名・会社名、送信日時、流入元メモ
  - 生成ログ（任意）：入力サマリ、生成結果のハッシュまたは全文、作成日時
- [ ] 接続プーリング／サーバーレス向け設定をドキュメント化

---

## 4. ドメイン（企業サブドメイン）

- [ ] 掲載サービス側と **サブドメイン名** を合意（例：`tool.example.jp`）
- [ ] Vercel の「Domains」にサブドメインを追加し、表示される **DNS レコード** を IT に依頼
- [ ] 検証：HTTPS で LP が開くこと

---

## 5. メール（Resend）

- [ ] Resend で **送信ドメイン**（上記サブドメイン or 親ドメイン）を追加
- [ ] Resend が提示する **SPF / DKIM 等の DNS** を企業 DNS に登録（IT 依頼）
- [ ] アプリ側：`RESEND_API_KEY` があるときだけ送信（未設定時は DB のみ・Slack のみでもリリース可）
- [ ] トランザクションメール：受付確認 1 通（件名・本文テンプレ、入力要約）
- [ ] マーケメールは受付確認と分離（必要なら別オプトイン）

---

## 6. AI（Gemini）

- [ ] サーバー側 API のみで `GEMINI_API_KEY` を使用（クライアントに露出しない）
- [ ] プロンプト：Markdown 見出し・太字ルール、下書きである旨の注意（過大表現を避ける）

---

## 7. LP／コンバージョン

- [ ] ヒーロー：採用担当者の課題（書けない・刺さらない等）を明確に
- [ ] 入力フォームは短く、生成は結果画面で読みやすく表示（Markdown レンダリング）
- [ ] 主 CTA：掲載サービスへの誘導を **生成直後** に配置
- [ ] プライバシーポリシー・同意（リードフォーム）
- [ ] **コピーライト表記**：フッター等の `©` 表記を、実在する運営主体・求人掲載サービス名・年度表記に合わせて修正（プレースホルダー・誤記・他社名の混入を排除）
- [ ] 計測：GA4 イベント例 `generate_success` / `lead_submit`、キャンペーン用 **UTM** 運用ルール（任意）
